# ADR 0009 — Accountant-ready PDF tax report

**Date:** 2026-05-31
**Status:** Accepted

## Context

[ADR 0006](0006-tax-engine-foundations.md), [0007](0007-belgian-tax-regime-estimates.md)
and [0008](0008-belgian-tax-declaration-mapping.md) put the full picture on
the Tax page: realized gains in EUR, three regime scenarios, the codes and
amounts that go on the Belgian declaration. The conversion lever the strategy
points at is still one step away: the file the user emails their accountant.

A one-page PDF report — same data the user sees on screen, in a form an
accountant can read in 30 seconds and a user can save to MyMinfin's
attachments — closes the arc. It is also the artefact that, with
`Verified by Koersa` in the footer, makes the product feel real on a
spreadsheet-driven workflow.

## Decision

### Dompdf as the renderer

- `dompdf/dompdf` (~2 MB Composer dep, MIT, actively maintained by community).
- Pure PHP, no binary in the container, works inside the existing PHP-FPM
  image with no Dockerfile change.
- CSS 2.1-class support — enough for the report (header, three or four
  stacked tables, footer); no flexbox or grid needed.
- Rules out:
  - **wkhtmltopdf via KnpSnappy** — wkhtmltopdf was archived upstream in
    2023, last release 2020. Picking an unmaintained binary at the heart of
    the product would be a long-tail tax-on-future-us.
  - **Mpdf** — more CSS coverage but ~10 MB, larger surface, no feature we
    need that Dompdf doesn't have.
  - **Headless Chromium (Puppeteer / Browsershot)** — best rendering, but
    adds Chrome to every container we ship. Massive for one-page output.

### A `Reporting` bounded context — first slice

[ARCHITECTURE.md §2](../../ARCHITECTURE.md) lists `Reporting` as a planned
context; this is the first slice that actually needs it.

The context introduces three files plus a template:

- `Reporting\Application\PdfRenderer` — port. `render(string $html): string`.
- `Reporting\Infrastructure\Pdf\DompdfPdfRenderer` — adapter behind the
  port. Future renderers (different lib, headless Chrome for a fancier
  template) drop in here without touching the call site.
- `Reporting\UI\Controller\TaxReportPdfController` — route
  `/tax/report.pdf`. Same `IsGranted` + organisation membership checks
  as the Tax page. Streams `application/pdf` with
  `Content-Disposition: attachment; filename="koersa-tax-report-{year}.pdf"`.
- `templates/reporting/tax_report.html.twig` — HTML with inlined CSS,
  handed to Dompdf as a self-contained document (Dompdf's
  external-stylesheet loader is fiddly across filesystems).

### `Reporting → Portfolio` allowed in deptrac

Reporting is the cross-context aggregator. By design it reads from
other contexts to compose reports; other contexts never read from it.
The deptrac config is updated accordingly:

```yaml
Reporting: [Shared, Portfolio]   # downstream of Portfolio
Portfolio: [Shared]              # Portfolio doesn't see Reporting
```

This is the correct DDD shape (downstream / upstream relationship) and
matches the ARCHITECTURE.md intent. As more reports land that need data
from IAM, MarketData, etc., the rule grows additional contexts on the
right-hand side; the inverse stays empty forever.

The compromise sketched earlier for `OverviewController` (keeping it in
Portfolio because Shared can't reach into a context) does not apply
here — Reporting is *allowed* to reach in; Shared still isn't.

### What goes in the report (one page, in this order)

1. **Header** — Koersa wordmark, the title "Belgian tax report — income
   year YYYY", the account email and organisation, generation timestamp.
2. **Summary** — total realized gain, total proceeds, total cost basis;
   if any unmatched sells, the count plus a one-line note.
3. **Realized gains per asset** — asset, proceeds, cost basis, gain.
4. **Regime scenarios** — three rows: regime / rate / potential tax.
5. **Filing guidance** — three rows: regime / Cadre + codes / amount to
   enter (the slice-3 mapping).
6. **Methodology** — FIFO, ECB reference rates on the trade date, the
   disclaimer that codes can shift and Koersa shows scenarios not advice.

### Locale follows the user

The PDF is rendered in the locale the user is currently in (via the
existing `LocaleSubscriber` — same translation catalogues that drive the
Tax page). A user reading the Tax page in French gets an FR PDF; same for
NL and EN.

### `Verified by Koersa` in the footer

Each page footer carries a small `Generated by Koersa · koersa.be · {date}`
line. Establishes the artefact's origin without claiming authority over
the tax outcome.

## Out of scope (this slice)

- **Multi-page reports.** One page is enough for the data we have. Larger
  portfolios will wrap; ship that when a real user hits the cap.
- **Visual branding** beyond the wordmark and footer. A designer pass
  comes later, when there's a designer.
- **Cryptographic signing** of the PDF for authenticity. Belgian filing
  doesn't require it; Koersa isn't a signed-statement authority.
- **Per-asset trade breakdown** (lot-by-lot, every leg with date and
  rate). Real auditability eventually wants this — out for the first
  ship; the user has the underlying data in the app and can export
  separately if needed.
- **Year picker.** The report year matches `forMostRecentYear()` —
  same year the Tax page shows. A picker comes when someone has more
  than one filing-relevant year of activity.
- **API endpoint** returning the same data as JSON (would be cheap, no
  product reason yet).

## Consequences

- One new Composer dep (`dompdf/dompdf`). Pure PHP, no system-level
  install; the existing PHP image already runs it. Acceptable per the
  ARCHITECTURE.md proposal rule because (a) it's pure PHP, (b) it's the
  smallest viable PDF library, (c) it's load-bearing for the conversion
  lever and the alternative is hand-rolled PDF emission.
- The `Reporting` bounded context exists as soon as this lands. The
  three other contexts (Billing, Integration, Reporting) listed in
  ARCHITECTURE.md but not yet built remain not-yet-built; this slice
  takes only the one we need.
- The "scenarios not advice" stance from ADR 0007 / 0008 carries into
  the PDF unchanged: the report shows what the user could owe under
  each regime, points at the codes, and disclaims.
